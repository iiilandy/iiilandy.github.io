开发基础

1、现代化的Web开发思想

前后端分离，前端工程化，后端微服务化。

2、从CS到BS的思路转换

传统的windows单体应用程序，是一种非中心化的架构。程序所消耗的资源是程序宿主机的资源，每一个客户端都在独立的应用程序域中，一般没有服务端程序，而是直接连接到一个数据库服务来检索和持久化数据。性能问题一般是：
1. 客户端宿主机配置导致运行慢
2. 关系型数据库查询慢
3. 持久化数据时的磁盘IO慢

客户端架构中，一般大量使用存储过程，这样可以减少频繁更新发布产品的工作量。
但也有多种坏处：
1. 业务逻辑分散在代码和存储过程中，给定位bug带来障碍。
2. SQL语言可读性差，需要花费更多时间去搞明白
3. 数据库服务器需要耗费更多资源来承担业务逻辑运算。

基于BS架构的现代化Web应用程序，是一种中心化的架构。所有使用者通过浏览器上的前端Web网站来进行使用（和后端服务进行交互），使用者越多，服务端资源占用就越多。性能问题除了数据库瓶颈，还会发生在代码中，所以要注意代码的质量。

boot-bom工程

BOM（Bill of Materials）定义一整套相互兼容的jar包版本集合，使用时只需要依赖该BOM文件，即可放心的使用需要的依赖jar包，且无需再指定版本号。BOM的维护方负责版本升级，并保证BOM中定义的jar包版本之间的兼容性。

为什么需要BOM？

使用BOM除了可以方便使用者在声明依赖的客户端时不需要指定版本号外，最主要的原因是可以解决依赖冲突。

考虑以下的依赖场景：

Project A依赖B 2.1.3和C 1.2.0版本
B 2.1.3依赖D 1.1.6版本
C 1.2.0依赖D 1.3.0版本

在上例中，Project A对于D的依赖就出现冲突，按照maven dependency mediation的规则，最后生效的是1.1.6版本（就近原则）。
在这种情况下，由于C依赖1.3.0版本的D，但是在运行时生效的确是1.1.6版本，所以在运行时很容易产生问题，如 NoSuchMethodError, ClassNotFoundException等。

为了统一各个spring-boot工程中jar包的版本，创建了了boot-bom工程，其他工程导入boot-bom，确定jar包版本。

jbase工程

代码目录：$/基层研发中心/070医养融合/00.公共应用/05.程序源码/iot/server/iot
物联网后端，根据当前需求拆分了4个模块出来，分别是 common、security、api、core。

  • common模块中定义、提供了整个工程常用的实体类、工具类等，提供给工程的其他模块调用。
  • security 模块中配置了安全校验、登录授权。
  • business 模块中实现了通用的基础业务，主要是查询为主，例如病区、科室、人员获取。

boot-demo-parent

代码目录: $/基层研发中心/070医养融合/00.公共应用/05.程序源码/demo/boot-demo-parent

  • admin spring boot admin server 模块，用户监控微服务运行状况
    • 启动后访问：http://localhost:8084/spring-boot-admin-server/
  • common 模块定义工程内通用工具、异常、自定义事件等
  • core 核心业务模块，各业务实现代码
  • dubbo dubbo 调用示例，其中有三个子模块分别是：
    • dubbo-common 公用模块，制定接口标准
    • provider 实现接口标准
    • consumer 消费接口
  • schedule 模块为Kob作业客户端，可单独启动，不影响主业务
    • Kob 控制台示例：http://172.16.0.91:8669/
